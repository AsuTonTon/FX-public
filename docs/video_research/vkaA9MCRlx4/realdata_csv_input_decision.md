# 実データCSV投入前の仕様固定

## 1. 目的

この文書は、Strategy A に実データCSVを投入する前に、データ仕様と検証前提を固定するためのものです。

目的は、読み取り専用バックテストの入力仕様を明確にし、後から都合よく前提を変えないようにすることです。実運用判断、発注判断、収益性判断を行う文書ではありません。

この文書では以下を扱いません。

- 実データ取得API
- ブローカーAPI
- 自動売買
- 発注
- 実資金
- Strategy B/C
- 収益性の断定

## 2. 使用候補データソース

初回検証では、手元に用意したローカルCSVのみを使います。

使用候補:

- ローカルCSV
- TradingViewからの手動エクスポート
- Dukascopy等の外部ヒストリカルデータを手動でCSV化したもの
- その他、OHLCをCSVとして手動で用意できるデータ

今回の作業では、データ取得スクリプトやAPI連携は実装しません。データ取得元がどこであっても、Strategy A runner はローカルCSVを読み取るだけです。

## 3. CSV必須仕様

必須カラム:

| カラム | 内容 |
|---|---|
| `timestamp` / `time` / `datetime` / `date` | 足の時刻 |
| `open` | 始値 |
| `high` | 高値 |
| `low` | 安値 |
| `close` | 終値 |

任意カラム:

| カラム | 内容 |
|---|---|
| `volume` | 出来高。現行 runner では使用しない |

CSV仕様:

- 文字コードは `UTF-8` または `UTF-8 BOM`。
- 1ファイル1通貨ペア。
- 推奨ファイル名は `PAIR_4h.csv`。例: `USDJPY_4h.csv`。
- 古い足から新しい足へ昇順。
- 4時間足の確定済みデータのみ。
- 未確定足は入れない。
- 日足CSVや1時間足CSVを4時間足として流用しない。

現行 validation では以下を拒否します。

- 重複 timestamp
- 欠損値
- timestamp 昇順違反
- `high < low`
- `high < open` または `high < close`
- `low > open` または `low > close`
- timezone aware と naive の混在
- 4時間足が主ではないCSV

## 4. タイムゾーン方針

推奨方針:

- 可能なら UTC に統一する。
- UTCに統一できない場合は、取引プラットフォーム時刻をそのまま使い、タイムゾーンを明記する。
- runner 内ではタイムゾーン変換をしない。

初回検証で固定する項目:

| 項目 | 方針 |
|---|---|
| 推奨タイムゾーン | `UTC` |
| UTCに統一できない場合 | 取引プラットフォーム時刻を採用し、タイムゾーンを記録 |
| timestamp の意味 | 原則として足の開始時刻。異なる場合は明記 |
| 夏時間の扱い | データ提供元仕様を記録 |

4時間足の区切り時刻は、データソースごとに固定して記録します。

例:

- `00:00 / 04:00 / 08:00 / 12:00 / 16:00 / 20:00 UTC`
- または、ブローカーサーバー時刻ベースの4時間区切り

どちらを使う場合でも、同じ検証内で混在させません。

## 5. 価格種別

候補:

- Bid
- Ask
- Mid
- broker close

初回検証の推奨:

1. Mid OHLC が明確に取得できるなら Mid を採用。
2. Mid がない場合は、Bid OHLC または broker close を採用し、価格種別を明記。
3. Ask OHLC を使う場合は、その理由を記録。

初回検証で記録する項目:

| 項目 | 記録値 |
|---|---|
| 採用価格種別 | `TBD: Mid / Bid / Ask / broker close` |
| open/high/low/close が同じ価格種別か | `TBD` |
| broker close の定義 | `TBD` |

Bid/Askがない場合の限界:

- 実際の約定価格との差は検証に含まれない。
- スプレッドの時間変動は再現できない。
- TP/SL到達判定が実売買と一致するとは限らない。
- 結果は「指定OHLC上の機械的検証」であり、実運用可能性を示さない。

## 6. スプレッド・コスト前提

現行実装で扱う項目:

- `assumed_spread`
- `tick_size`
- `buffer`

現行実装で直接扱わない項目:

- 手数料
- スリッページ
- 約定拒否
- 可変スプレッド

初回検証で固定する方針:

- `assumed_spread` はペア別に固定値を置く。
- `tick_size` はペア別に固定する。
- `buffer` は原則 `assumed_spread + tick_size` とする。
- 手数料は初回検証では `0` として扱う。ただし制約として明記する。
- スリッページは初回検証では `0` として扱う。ただし制約として明記する。

後続で感度分析すべき項目:

- `assumed_spread`
- `buffer`
- 手数料
- スリッページ
- 同一足TP/SL到達時の扱い
- 週末またぎの取引除外

注意:

`config/strategy_a_pairs.example.json` の値は仮定値です。現在の実スプレッド、ブローカー提示値、推奨値ではありません。

## 7. 週末ギャップ・欠損・異常値

週末ギャップ:

- 週末ギャップは許容する。
- 4時間足CSVで、8時間以上の差分は週末・休場ギャップ候補として許容する。
- ただし、4時間足が主でないCSVは拒否する。
- 日足CSVは拒否する。

欠損・重複:

- 重複 timestamp は拒否。
- 欠損 timestamp は拒否。
- 欠損 OHLC は拒否。
- timestamp が昇順でないCSVは拒否。

OHLC不整合:

- `high < low` は拒否。
- `high < open` / `high < close` は拒否。
- `low > open` / `low > close` は拒否。

異常値:

- runner 内で自動補正しない。
- 明らかな異常ヒゲ、異常スパイク、長期欠損は手動確認する。
- 修正する場合は、元データ、修正理由、修正箇所を別途記録する。
- 合成足や補完足を無断で入れない。

## 8. 初回検証対象ペア

初回は2ペアに絞る案を採用します。

初回対象:

- `USDJPY`
- `EURUSD`

理由:

- メジャー通貨ペアでデータを用意しやすい。
- JPY系と非JPY系を1つずつ比較できる。
- まずCSV品質、イベントログ、runner出力の妥当性を見る段階なので、対象を広げすぎない。

次の拡張候補:

- `GBPUSD`
- `AUDUSD`
- `USDCAD`
- `USDCHF`
- `EURJPY`
- `GBPJPY`

最初から8ペアにしない理由:

- データ品質確認の負荷が大きくなる。
- ペアごとのスプレッド、tick_size、価格桁、週末ギャップ仕様の確認が雑になりやすい。
- 出力結果が多くなり、Strategy Aの核の挙動確認より集計確認に寄りやすい。

サンプル数不足のリスク:

- 2ペアだけでは、手法全体の一般性は判断できない。
- 取引回数が少ない場合、勝率・PF・total R は不安定になる。
- 初回検証は収益性評価ではなく、データ投入とルール挙動の確認と位置づける。

## 10. 次の作業

1. ユーザーが実データCSVを用意する。
2. CSVを `data/strategy_a_ohlc/` に配置する。
3. ファイル名を `USDJPY_4h.csv`, `EURUSD_4h.csv` のように揃える。
4. `config/strategy_a_pairs.example.json` を必要に応じて検証用設定としてコピーし、`tick_size`, `assumed_spread`, `buffer` を固定する。
5. runnerを読み取り専用で実行する。
6. 出力された `summary.md`, `metrics`, `trades`, `events` を確認する。
7. Strategy A の核に検証価値があるか判断する。

確認観点:

- シグナル数が極端に少なすぎないか。
- EMAウォームアップ後だけでsetupが始まっているか。
- 4時間足validationで想定外のギャップが出ていないか。
- entry / stop / take profit が仕様通り出ているか。
- end_of_data 決済が多すぎないか。
- 結果を収益性の証明として読んでいないか。
